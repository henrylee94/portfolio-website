01 — signature work

Signature Work

Case 01

Workflow backbone — 8.4M audited workflows a day

Situation

Every money-moving action on the platform — withdrawal, top-up, reward, configuration change — must pass one audited create → review → approve → execute lifecycle. The legacy implementation was tightly coupled inside a monolith; every consumer re-implemented its own wiring.

Constraint

This is the platform's single point of trust: production processes ~8.4M workflow records a day, and a duplicate execution means real money moved twice.

Approach

Rebuilt it as a gRPC microservice orchestrated by Temporal: one generic durable workflow runs every request type, with human review modeled as signals (30-day timeout guards) and business extension points as child-workflow slots. Idempotency is structural — the workflow ID doubles as the creation key, so a retried step returns the existing record instead of creating twice. Integration became a published SDK: any service registers an executor with one import; adding a new request type is a self-service CI form, not a queue on my desk. 100+ SDK releases, SemVer-disciplined.

Outcome

8.4M records/day in production. Duplicate execution is prevented by construction rather than caught after the fact. New-type integration went from a 30-to-60-minute, person-dependent process to two-minute self-service.

Case 02

Scaling a platform 50k → 300k+ users

Situation

Over four years the platform grew 6×. Old patterns failed in production more than once as limits were hit — a counter document in the database serialized every ID; notification fan-out stalled under load.

Constraint

No rewrite budget, no downtime allowance — every fix had to land inside a weekly release train, live, with users on.

Approach

Root-caused each bottleneck to its mechanism, then replaced the mechanism, not the symptom: ID generation moved from a database counter to atomic Redis INCR — cut over with a +1-billion offset so old and new ranges could never collide; notification fan-out moved to Redis pub/sub (shipped, reverted on a production surprise, re-landed hardened); blocking full-set scans replaced with cursor-based SSCAN; a leaking in-memory config cache rebuilt with a 15-minute refresh cycle.

Outcome

The platform sustains 300k+ DAU without the old failure modes. The patterns introduced (cache-aside, atomic counters, pub/sub fan-out) became house standards for modules built after.

Case 03

Real-time support platform — from zero to clean handoff

Situation

The company needed AI-assisted customer conversations with human takeover — and had just decided on a new microservices stack no one had shipped to production yet. I was handed both problems at once.

Constraint

Solo start; NestJS, gRPC, Temporal, Pulsar and Matrix — all new to me at once; the outcome would decide whether the company's migration was viable.

Approach

Self-hosted Matrix/Synapse as the message transport (the framework was the CTO's call; the data model and flows were mine), then built everything it doesn't have: an agent-routing queue encoding status > load > wait-time into a single Redis sorted-set score (three numeric bands: 1e18 status, 1e15 active-chat count, millisecond timestamp tie-break — assignment is O(1) at the queue head); a mini back-office so agents see a user's records, history and permissions in one screen instead of switching systems; AI answers first — designed to absorb the repetitive ~70% of inquiries — with a 10-second race guard that hands the chat to a human whenever the model stalls or errors. Concurrency was the war: a read-modify-write race on scores was rebuilt as atomic increments — and when production revealed the Redis cluster wouldn't take MULTI, the fix was degraded the same day to pure atomic ops; duplicate agent invites under concurrent messages were fenced with per-room locks.

Outcome

First production service on the new stack — the pattern reference (config via Vault, contract-first gRPC, structured logging, graceful shutdown) that later services copied. Grew the team to two, handed the service to a maintenance team, later to the AI team. It now runs as a standalone multi-channel middleware.

Case 04

Cross-stack mission pipeline — three languages, exactly-once

Situation

Marketing wanted automatic, time-boxed missions for users close to their next loyalty tier: detect them in real time, run a 7-day window, grant the upgrade instantly on success.

Constraint

The reward path is money-adjacent: a duplicate grant is a real loss, and events arrive at least-once from both the stream and the workflow engine — two independent sources of duplicates.

Approach

Built the chain end-to-end across four services in three languages — the boundaries followed the platform's existing seams (the streaming platform, the workflow engine, the reward facade), composing proven engines instead of building new ones: a Java/Flink job filters the event stream and computes entry-anchored goals from a lookup table; a Rust + Temporal engine runs one durable workflow per user; a NestJS facade grants through the existing upgrade path — zero changes to legacy systems. Exactly-once is layered: an atomic claims ledger (one chance per user, tier, month) gates enrollment, and a human-readable idempotency key (`mission:user:tier:month`) gates the grant — debuggable at a glance in production. Thresholds live in exactly one place; producers broadcast unconditionally, so no cross-repo value coupling.

Outcome

Verified end-to-end against real traffic in the pre-production lane; funnel metrics at every stage. The architecture — detect / orchestrate / grant with single-owner config — is the template for the next mission types.

Case 05

Transaction-record ingestion — 12–14M records a day from 10+ sources

Situation

Every balance change a user sees must be complete, ordered, and fast to query — but records originate from 10+ heterogeneous sources that disagree on shape, timing, and delivery guarantees: ~8.4M/day through the audited workflow path, the rest streaming directly from providers and internal systems.

Design

Split storage by read pattern: hot recent history on the OLTP store with covering indexes; long-window analytics offloaded to a columnar layer fed by the ingestion pipeline; source-specific shapes normalized at the boundary, with per-source caching and channel-aware routing.

Outcome

12–14M records/day ingested and queryable; recent-history reads stay fast while analytical queries stopped competing with production traffic.

Case 06

Crypto payment rails — two years, three wallet protocols

Situation

The platform needed cryptocurrency deposits and withdrawals alongside fiat — conversion at every transaction boundary, on a platform where some third-party providers couldn't hold crypto at all.

Design

Drove the money-side across two years as one of a two-engineer pair (alongside the team's assistant lead): wallet protocol integrations (OMNI, ERC-20, TRC-20), conversion embedded at transfer boundaries, provider capability as per-provider configuration — and exact-decimal handling at the wallet edges, where rounding errors become real discrepancies.

Outcome

Crypto rails went live across every money flow; later currencies reused the same config-driven paths instead of re-opening them.

Case 07

Reporting engine & regulatory delivery

Situation

Before a BI team existed, operations, partners, and management all needed data — and later, a regulator needed daily filings in exact formats: millions of rows per day, hard deadline, aggregated over long windows on the production database.

Design

Built the platform's general-purpose reporting layer on a precompute + recompute dual track: an independent settlement process streams per-user transaction flows to workers in batches, landing daily summaries; ad-hoc queries run tuned aggregation pipelines. Rewrote the scheduler for reliable 2-day recompute windows. For regulatory delivery, built the filing pipeline: million-row-per-file slicing, append-based FTP upload, stream batches tuned until multi-million-row days fit the window.

Outcome

The platform had a working BI function before it had a BI team — the formal team later built on these foundations. Regulatory filings shipped on schedule at million-row scale; 140+ reporting changes over six years.

02 — how i think

How I Think

Principles first, tasks second. These are the questions I ask before anything gets built.

P1

Price the risk before pricing the work.

The first question is never "how long will it take" — it's "what breaks if this is wrong, and who feels it?" Failure cost decides engineering rigor: money-path modules get idempotency by design, staged rollouts and pre-launch index reviews; a cosmetic feature gets speed instead.

Example: our workflow backbone carries every money-moving action — so retries are idempotent by construction and shutdowns drain in-flight work before the pod dies.

P2

Design for the next change, not this delivery.

Requirements on this platform change weekly. In every design review I ask which parts business will want to tune later — those become configuration, not code. A feature request should become an ops change, not a deploy.

Example: when we added a new settlement currency, provider compatibility was config per provider — incompatible ones toggled off, rollout reversible by a flag.

P3

Decompose by ownership, not by ticket.

Big goals get split along module boundaries with explicit contracts — proto first, then parallel build. Each engineer owns one domain end-to-end; one floating engineer absorbs surges. People grow by owning, not by taking tasks.

Example: a cross-cutting feature landed across five repositories in one day because the contracts were split before anyone wrote code.

P4

Decide with evidence, commit with a rollback.

Big technical choices get benchmarks and POCs, not opinions — and every migration ships with an escape hatch. If we can't measure it or can't undo it, we're not ready to decide.

Example: an OLAP store evaluation ran 30 queries × 50 rounds across three engines; when JSON-path performance failed the bar, we flattened the schema to remove the weakness — then re-measured.

P5

Turn work into leverage.

A bug fixed is a cost; a bug turned into a playbook or a bot is an asset. Anything done twice becomes a tool: integration SDKs, incident playbooks, an AI diagnosis bot the whole team runs.

Example: one silent contract-mismatch incident became a 7-step tracing playbook; recurring production triage became our AI "Bug Doctor" wired into MCP and Lark.
